- split panes should have their own state for thread visibility, etc., unaffected by actions in other panes. The core purpose of split panes is so a user can participate in different threads in the same chat without having to scroll a single window back and forth. So the UX needs to empower that use case. Panes should act like independent "slices" of the chat. The underlying chat state (incoming/outgoing messages, etc.) is centralized, but each pane "view" has it's own scroll (already imlpemented), input focus (already implemented), thread visibility state, etc. This also means the top (chat name, members online, etc.) and bottom (keyboard shortcuts, command menu) "menu" bars should be top-level, outside the panes (currently they are repeated for each pane)
- submitting a message in subthread while in split pane causes focus to go to end of top-level thread
- `sqwok join <INVITE_CODE>` from a new user should run onboarding/key generation if it doesn't already, then bring user to chat
- `v 3 new` messages "scroll down" indicator counts all messages as "new"?
